## ★次タスク: cross-exchange lead-lag 検証

### データ収集状況

- **bitFlyer FX_BTC_JPY**: Mac で `scripts/collect_bitflyer.py` がバックグラウンド実行中 (PID 27113)
  - 出力: `scripts/data_cache/bitflyer_ticker_2026-04-10.csv`
  - 3 秒間隔で bid/ask/mid を記録
  - Mac スリープで停止する → 起動時に再実行が必要
- **GMO metrics**: VPS bot (v0.14.4) が 3 秒間隔で metrics.csv を記録中

### 検証方法

1. bitFlyer と GMO の concurrent mid_price を 3 秒間隔で取得（数時間〜1 日分）
2. cross-correlation を計算: `corr(bf_return(t), gmo_return(t+lag))` for lag = 0, 1, 2, ... ticks
3. lag > 0 で有意な相関 → GMO が bitFlyer に遅れて動く → lead-lag alpha
4. 取引可能性: edge magnitude > GMO の half-spread (実測 ~600 JPY) か?

### 成功基準

- cross-correlation の t > 3.0 at lag >= 1
- edge > GMO half-spread (~0.6 JPY per 0.001 BTC)

### lead-lag が見つからない場合

GMO BTC/JPY でリテール bot の収益化は困難と判断。選択肢:
- 学習目的として受容
- 別の市場/アセットを検討
- 資金を増やして FR 裁定（Bitget）

## 重要な教訓（memory に記録済み）

1. **戦略検証を実装の前に** — Python 1 日で検証 → エッジ確認 → Rust 実装
2. **in-sample で判断しない** — 必ず train/test 分割、validation は 1 回のみ
3. **パラメータ最適化でエッジは生まれない** — エッジがない戦略は何を調整しても儲からない
4. **spread はシグナル時に拡大する** — 平均 spread で試算すると楽観的すぎる
5. **バックテスター精度 ≠ 戦略品質** — 正確なバックテスターが break-even を正確に報告するだけ
